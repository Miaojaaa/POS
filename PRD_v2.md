# เอกสารข้อกำหนดผลิตภัณฑ์ (PRD) - เวอร์ชัน 2.0
## ระบบแพลตฟอร์มบริหารจัดการธุรกิจบิวตี้และค้าปลีกแบบครบวงจร (Multi-Tenant Beauty, Wellness & Retail OS)

**เวอร์ชัน:** 2.0  
**วันที่:** 29 พฤษภาคม 2026  
**สถานะ:** Draft / Proposed Architecture  

---

## 1. ภาพรวมของระบบ (System Overview)

### 1.1 วัตถุประสงค์
ยกระดับจากระบบบริหารจัดการร้านเสริมสวยเดี่ยว (Single-Salon POS) สู่ระบบ **SaaS Platform แบบ Multi-Tenant** ที่รองรับลูกค้าร้านค้าหลายธุรกิจในกลุ่ม **Beauty, Wellness และ Retail** โดยมีจุดเด่นในด้านความยืดหยุ่นของหน้าตาการใช้งาน (Preset Layouts) สิทธิ์การเข้าถึงแบบละเอียด (Discord-like Custom RBAC) และระบบคำนวณทางการเงิน/คลังสินค้าที่มีเสถียรภาพสูงสุดระดับ Enterprise เพื่อป้องกันจุดบกพร่องด้านบัญชีและระบบสต็อก

### 1.2 สถาปัตยกรรมหลักและการป้องกันจุดบอด (Architectural & Safety Principles)
เพื่อหลีกเลี่ยงสภาวะ "ระบบพังเพราะ Logic ชนกันเอง" แพลตฟอร์มนี้จะยึดหลักการออกแบบดังนี้:
1.  **Unified Commerce Model:** ยุบรวมบริการ (Service) และสินค้า (Product/Retail) เข้าเป็นตารางแกนกลางตารางเดียว (`CatalogItem`) เพื่อให้ระบบคำนวณและตัดสต็อกใช้ Logic ชุดเดียวกัน
2.  **Immutable Financial Truth Layer:** ตัวเลขทางการเงิน ภาษี และค่าคอมมิชชันพนักงานในอดีตทั้งหมดจะถูก **Freeze (Snapshot)** ลงรายการธุรกรรมทันทีที่จบการขาย ห้ามใช้วิธีดึงสูตรคำนวณแบบ Dynamic ย้อนหลังเด็ดขาด
3.  **Event-Based Stock Ledger:** เลิกใช้วิธีการลบจำนวนสต็อกคงเหลือตรงๆ บนตารางสินค้าตัวเลขเดี่ยว แต่ใช้การเขียนธุรกรรมบันทึกความเคลื่อนไหวสต็อกเป็นแถวธุรกรรมใหม่เสมอ (`StockMovement`) ป้องกันปัญหาสต็อกเพี้ยนหรือ Race Condition
4.  **Enforced Tenant Isolation:** แยกข้อมูลระหว่างเจ้าของร้านค้า (Tenant) โดยใช้ฟิลด์ `tenantId` ครอบคลุมทุกตาราง ร่วมกับการตั้งค่า Row-Level Security (RLS) ที่ระดับ PostgreSQL และการตั้ง Cache Keys ที่มี Tenant Prefix เพื่อป้องกันข้อมูลลูกค้าร้านค้ารั่วไหลหากัน

## 2. แผนการเปลี่ยนฐานข้อมูลเป็น PostgreSQL (PostgreSQL / Prostege SQL Database Migration)

จากระบบเดิมที่ใช้ **SQLite** ซึ่งเป็น Serverless Database เขียนอ่านไฟล์ในเครื่อง แพลตฟอร์ม SaaS นี้จะถูกปรับเปลี่ยนไปใช้งาน **PostgreSQL (Prostege SQL)** เต็มรูปแบบเพื่อความปลอดภัย ความเสถียร และประสิทธิภาพในการทำงานพร้อมๆ กันของอุปกรณ์หน้าร้านจำนวนมาก

### 2.1 สเต็ปการเปลี่ยนระบบฐานข้อมูล (Migration Roadmap)
1.  **แก้ไข Prisma Schema Config:**
    *   เปลี่ยนค่า `provider = "sqlite"` ไปเป็น `provider = "postgresql"` ในไฟล์ `schema.prisma`
    *   แก้ไขชนิดข้อมูลเฉพาะตัวของ SQLite เช่น การเปลี่ยน String Datetime ให้เป็น Native `DateTime` ของ PostgreSQL เพื่อความถูกต้องของการสืบค้นช่วงเวลา
2.  **ตั้งค่า Connection Pool & Environment:**
    *   เชื่อมต่อฐานข้อมูลผ่าน Connection String ใน `.env` ตัวอย่าง: `DATABASE_URL="postgresql://username:password@host:port/database_name?schema=public&pgbouncer=true"`
    *   เปิดใช้ pgBouncer เพื่อควบคุมและจำกัดการเปิดใช้ Connection ของพนักงานหน้าร้านและหน้าผู้จัดการจำนวนมาก
3.  **การย้ายข้อมูลประวัติเดิม (Data Migration):**
    *   สร้างสคริปต์ส่งออกข้อมูล (Export Script) จากไฟล์ SQLite ปัจจุบันให้ออกมาเป็นไฟล์ JSON/CSV
    *   นำเข้าข้อมูลเข้าสู่ PostgreSQL โดยการจัดเรียงลำดับตารางตามความสัมพันธ์ (Dependencies) ป้องกัน Foreign Key Constraint Error
4.  **การทำ Row-Level Security (RLS) ใน PostgreSQL:**
    *   สร้างนโยบายความปลอดภัย RLS ในทุกๆ ตารางหลักที่มีฟิลด์ `tenantId`
    *   คำสั่ง SQL เพื่อตั้งนโยบายความปลอดภัย:
        ```sql
        ALTER TABLE "Order" ENABLE ROW LEVEL SECURITY;
        CREATE POLICY tenant_isolation_policy ON "Order" 
        USING (tenant_id = current_setting('app.current_tenant_id', true));
        ```
    *   เมื่อ Prisma ดำเนินการ Query ระบบจะทำการเชื่อมโยง Session Context และระบุ `app.current_tenant_id` ในธุรกรรมอัตโนมัติ เพื่อป้องกันโอกาสเกิดข้อผิดพลาดในการดึงข้อมูลข้ามร้านค้าโดยมนุษย์ (Human Error Data Leak)

## 3. สถาปัตยกรรมการ Custom ระบบแบบละเอียด (Detailed Customization Specification)

ระบบแพลตฟอร์มนี้จะไม่เปิดสิทธิ์ให้ทำการ Custom แบบไร้ขอบเขต (No-Code Builder) เพื่อป้องกันปัญหาระบบทำงานขัดแย้งกันเอง แต่จะเปิดใช้การ **Custom ภายใต้ขอบเขตที่ควบคุมความถูกต้องทางการเงินและคลังสินค้าได้ (Opinionated Bounded Customization)**

### 3.1 การ Custom UI หน้าจอ POS (Layout Customization)
ผู้ใช้สามารถเปลี่ยนธีมและสไตล์หน้าจอยอดขายหน้าร้านได้ตามความถนัดของแต่ละธุรกิจ ผ่านระบบ Preset Slots และ Custom Display Options:

*   **Customization Options ที่ผู้ใช้ตั้งค่าได้:**
    *   **Layout Presets:**
        1.  *แบบปุ่มรูปภาพขนาดใหญ่ (Large Button Grid)* - เหมาะกับร้านอาหาร คาเฟ่ และร้านนวด
        2.  *แบบแถวตารางเน้นปริมาณข้อมูล (Compact List Grid)* - เหมาะกับร้านของชำและมินิมาร์ท
        3.  *แบบแบ่งสัดส่วนมาตรฐาน (Split Service Grid)* - เหมาะกับร้านสปาและเสริมสวยที่มีคิวจอง
    *   **Display Items Customization:**
        1.  *แบบโชว์เฉพาะข้อความ (Text-Only Mode)* - เหมาะสำหรับหน้าจอที่ต้องการความรวดเร็วและใช้การสแกนบาร์โค้ดเป็นหลัก
        2.  *แบบโชว์รูปภาพสินค้า (Image Grid Mode)* - เหมาะสำหรับร้านอาหารหรือบริการทำเล็บเพื่อแสดงตัวอย่างผลงาน
    *   **Orientations & Layout Types:** สามารถเลือกการแสดงผลเป็นแนวนอน (Landscape สำหรับแท็บเล็ต) หรือแนวตั้ง (Portrait สำหรับเครื่องมือถือสมาร์ทโฟนพกพา)

```
[ Layout Customizer Engine ]
     ├── Read Tenant Business Setup (e.g. Type: Salon, Layout: Compact List, Display: Text-Only)
     ├── Map Items to UI Slots -> [SLOT: MAIN_PANEL] -> Render: TextOnlyListRow
     └── Map Checkout Panel -> [SLOT: RIGHT_CHECKOUT] -> Render: QuickCheckoutTotal
```

### 3.2 การ Custom ระบบ Add-on ในสินค้าและบริการ (Add-on Customization Engine)
เพื่อตอบโจทย์ร้านค้าที่ต้องการส่วนเพิ่มพิเศษ (เช่น ร้านอาหารต้องการให้ลูกค้าเพิ่มไข่ดาวหรือเลือกความพิเศษ, ร้านสปาต้องการทำทรีตเมนต์อัปเกรด):

*   **การตั้งค่า Add-on (Configuration):**
    *   **Add-on Groups:** จัดกลุ่ม Add-on (เช่น กลุ่ม "เครื่องเคียง", กลุ่ม "สารเคมีเกรดOrganic")
    *   **Item Mapping:** ผูก Add-on Group เข้ากับสินค้าหรือบริการหลัก
    *   **Stock Attachment:** Add-on แต่ละตัวสามารถตั้งค่าระบุรหัสสินค้าปลีกเพื่อไปทำหักสต็อกได้ (เช่น Add-on "ทรีตเมนต์Organic" จะหักสต็อกผง Organic ในคลัง 20 มล.)
*   **ตารางสถาปัตยกรรม Add-on:**
    ```prisma
    model AddonGroup {
      id             String   @id @default(cuid())
      tenantId       String
      name           String   // e.g. "สปาอัปเกรด", "เครื่องเคียงอาหาร"
      
      addons         Addon[]
      catalogItems   CatalogItemAddonGroup[]
    }

    model Addon {
      id             String   @id @default(cuid())
      addonGroupId   String
      name           String   // e.g. "เพิ่มไข่ดาว", "ทำทรีตเมนต์พรีเมียม"
      price          Float    @default(0) // ราคาบวกเพิ่มพิเศษ
      linkedProductId String? // หากต้องหักสต็อกสินค้าในคลัง
      
      addonGroup     AddonGroup @relation(fields: [addonGroupId], references: [id])
    }
    ```

### 3.3 การ Custom บทบาทพนักงาน (Discord-like Custom Roles Engine)
ระบบยกเลิกข้อจำกัดที่ต้องผูกบทบาทไว้กับชื่อตายตัว แต่ใช้ตาราง **Dynamic Role Builder** เลียนแบบ Discord โดยบังคับใช้กฎหลักดังนี้:

*   **ข้อจำกัดเชิงสถาปัตยกรรม (Architectural Limits):**
    *   ต้องมีบัญชี **Owner เพียง 1 บัญชีต่อแบรนด์เสมอ** ไม่สามารถสร้างเพิ่ม ลบ หรือแก้ไขสิทธิ์ของ Owner ได้เพื่อความมั่นคงของสัญญาระบบคลาวด์
    *   พนักงาน 1 คนสามารถได้รับมอบหมายบทบาทการทำงาน (Roles) ได้มากกว่า 1 บทบาทพร้อมๆ กัน
*   **การจัดการสิทธิ์ผ่าน Permission Toggles:**
    ระบบมีตารางกำหนดสิทธิ์ในระดับบิต (Permissions List) เพื่อให้ Owner ปรับแต่งได้อย่างอิสระ:
    *   `VIEW_DASHBOARD` - สิทธิ์การเข้าดูรายงานยอดขายจริงและกำไรสุทธิ
    *   `EDIT_PRICE` - สิทธิ์การแก้ไขราคาขายที่หน้า POS
    *   `VOID_BILL` - สิทธิ์การยกเลิกบิลหรือคืนเงินออเดอร์ย้อนหลัง
    *   `EXPORT_PAYROLL` - สิทธิ์การเข้าถึงข้อมูลเงินเดือนพนักงาน
    *   `MANAGE_STOCK` - สิทธิ์การโอนย้ายสต็อกและการปรับปรุงคลังสินค้า
*   **Key PIN Configuration (ระบบ HR):**
    ในหน้าตั้งค่าสิทธิ์พนักงาน ผู้จัดการระดับสูงหรือเจ้าของร้านสามารถเข้ามากำหนดรหัส **PIN พิเศษ** ให้แก่พนักงานรายบุคคลหรือผู้ถือครอง Role นั้นๆ ได้ เพื่อใช้ใส่ยืนยันความรับผิดชอบเวลาแก้ไขราคาขายหน้าร้านหรือให้ส่วนลดพิเศษแก่ลูกค้า

### 3.4 การ Custom ค่าคอมมิชชันและผลตอบแทน (Custom Commission Engine)
สถาปัตยกรรมการคำนวณค่าคอมมิชชันจะไม่ผูกขาดด้วยคำสั่งตายตัว (Hardcoded logic) แต่ใช้ระบบ **Commission Rule Engine** ในการกรองและประมวลผลตัวเลขรายได้หลังหักต้นทุนเคมี:

*   **ทางเลือกการจ่ายค่าคอมมิชชันที่ผู้ใช้เลือกเปิดใช้งานได้:**
    1.  **Flat Rate Commission:** ค่าคอมคงที่ เช่น ได้รับ 20 บาทต่อการให้บริการ หรือได้รับ 10 บาทต่อยอดขายสินค้าปลีก
    2.  **Percentage Commission:** คิดเป็นอัตราร้อยละจากราคาสินค้าหรือบริการ
    3.  **Individual Item Commission:** กำหนดอัตราค่าคอมแยกเป็นรายสินค้าปลีกในตารางแคตตาล็อก (สินค้าชิ้นที่ 1 ได้รับ 50 บาท, สินค้าชิ้นที่ 2 ไม่ได้ค่าคอม)
    4.  **Role-Based Pool Split:** นำยอดเงินรายได้สะสมของแผนกมารวมเป็นเงินกองกลาง (Pool) แล้วนำยอดเงินนั้นมาเฉลี่ยหารแบ่งกันตามจำนวนพนักงานที่ลงทำงานในแผนกนั้นๆ
    5.  **Multi-Role Commision Logic:** หากพนักงานคนหนึ่งรับงานมากกว่า 2 roles ขึ้นไปในวันเดียวกัน (เช่น เป็นทั้งช่างใหญ่ทำสี และผู้ช่วยสระผม) ระบบจะคำนวณสัดส่วนค่าคอมโดยอ้างอิงจากบทบาทผู้รับผิดชอบงานที่ถูกทำ **Snapshot บันทึกเป็นชิ้นงานลงในแต่ละรายการสั่งซื้อย่อย (`OrderItem`) ณ เสี้ยววินาทีที่สแตมป์ปิดออร์เดอร์** เพื่อให้ยอดคำนวณใน Payroll สรุปผลได้อย่างถูกต้องเป็นธรรม

### 3.5 การ Custom ระบบคลังสินค้า (ERP Customization Mode)
เมื่อตั้งค่าระบบ ERP ใน onboarding เจ้าของร้านค้าสามารถเลือกตั้งค่าพฤติกรรมของคลังสินค้าตามขนาดและความเหมาะสมของธุรกิจ:

*   **ระบบคลังแยกอิสระ (Isolated Branch Stocks):**
    *   แต่ละสาขาสามารถทำใบสั่งซื้อ (PO) และนำเข้าสต็อกเข้าคลังส่วนตัวได้โดยตรง ไม่ผูกมัดกับสาขาอื่น
    *   ยอดต้นทุนรวมเฉลี่ยถัวเฉลี่ยเคลื่อนที่ (Moving Average Cost) จะถูกคำนวณแยกอิสระรายสาขา เหมาะสำหรับธุรกิจแฟรนไชส์
*   **ระบบคลังกลางและสาขาย่อย (Centralized Multi-Warehouse Stocks):**
    *   มีคลังสินค้าใหญ่เป็นศูนย์กลางในการรับสินค้าเข้า
    *   สาขาย่อยจะไม่สามารถนำสินค้าเข้าคลังได้เอง แต่ต้องทำใบคำขอโอนย้ายสต็อก (`StockTransferRequest`) เพื่อให้ผู้จัดการสาขาหรือผู้ดูแลคลังสินค้าใหญ่อนุมัติโอนสต็อกส่งสินค้าออกไปยังสาขาปลายทาง
    *   ประวัติการโอนสินค้าและความเคลื่อนไหวสต็อกจะถูกบันทึกรหัสอ้างอิงของเจ้าหน้าที่ผู้โอนและผู้อนุมัติด้วยคีย์ PIN ประจำตัวในระบบ Audit Logs เสมอ

## 4. แผนงานการส่งมอบและพัฒนา (Phased Roadmap)

ระบบจะถูกพัฒนาโดยหั่นแบ่งความสำคัญออกเป็น 4 เฟส เพื่อสร้างรากฐานที่มั่นคงก่อนขยายตัว:

```
[Phase 1: รากฐานข้อมูลและ Tenant Isolation]
  ├── ติดตั้ง PostgreSQL (Prostege SQL) + เชื่อมต่อ Prisma Client
  ├── ออกแบบ Tenant Isolation (RLS + Tenant Context ใน API)
  ├── โครงสร้างตาราง Unified CatalogItem และ Immutable Order Snapshot
  └── ระบบล็อกอินแบ่งลูกค้าร้านค้าหลัก
  
[Phase 2: UI Preset Layouts และ POS Core]
  ├── พัฒนา POS Cart Engine รองรับ Add-ons และ Display Options
  ├── ทำ Preset Slots UI (โหมดบิวตี้/โหมดร้านค้าปลีกสแกนบาร์โค้ด)
  └── ระบบ Ledger-based StockMovement (คลังสินค้าแบบ Event log)

[Phase 3: Discord-like Roles และ Custom Commission]
  ├── พัฒนา Role-Based Access Control แบบละเอียด (Dynamic Role Table)
  ├── ระบบ Re-authentication ในฟังก์ชันสำคัญ (ลบออเดอร์, คืนเงิน)
  └── ระบบคำนวณและ Snapshot ค่าคอมมิชชันพนักงานในออเดอร์

[Phase 4: Multi-Branch และ SaaS Automation]
  ├── ระบบ Wizard Onboarding อัตโนมัติสำหรับธุรกิจใหม่
  ├── ระบบโอนย้ายสต็อกข้ามคลัง/ข้ามสาขา และใบสั่งซื้อ (ERP Module)
  └── ระบบ Degrading Offline Mode (ขายต่อได้ชั่วคราวแม้อินเทอร์เน็ตล่ม)
```

## 5. ข้อกำหนดที่ไม่ใช่หน้าที่หลักและการรักษาความปลอดภัย (Non-Functional Requirements)

### 5.1 ระบบความปลอดภัยทางการเงินและการเงินย้อนหลัง (Re-auth Security Policy)
*   การเข้าดูหน้ารายงานสรุปความเคลื่อนไหว (Audit Logs) จะไม่ใช้วิธีการเด้งกล่องถามรหัสผ่านซ้ำๆ ซากๆ ทุกครั้งที่กดเปลี่ยนเมนู แต่จะใช้วิธีการกำหนดบทบาทตั้งแต่สิทธิ์การเปิดเพจ
*   **Re-authentication:** จะถูกทริกเกอร์ให้ทำรายการในระดับธุรกรรมที่มีความเสี่ยงสูงเท่านั้น โดยจะเด้งหน้าต่างกรอก PIN 4-6 หลัก หรือรหัสผ่านหลักในบราวเซอร์:
    1.  การกดยืนยันให้ส่วนลดบิลขาย (เมื่อรหัส PIN ของพนักงานที่ล็อกอินอยู่มีระดับสิทธิ์ไม่พอ)
    2.  การกดลบรายการสินค้าหรือ Void บิลย้อนหลัง
    3.  การปรับปรุงยอดคงเหลือในกระเป๋าเงินสมาชิก (Wallet Adjustments)
    4.  การกด Submit หรือออกรายงานเงินเดือนพนักงาน (Payroll Export)

### 5.2 ระบบกู้คืนสถานะการทำงานขณะเน็ตล่ม (Graceful Degrading Offline Mode)
*   **Local Caching:** บราวเซอร์ของเครื่อง POS จะทำการสำรองข้อมูลสินค้า หมวดหมู่สินค้า และค่าบริการต่างๆ ล่าสุดเอาไว้ใน LocalStorage/IndexedDB
*   **Offline Order Queue:** เมื่อสถานะเครือข่ายเปลี่ยนเป็นล่ม (Offline) ระบบ POS จะสามารถสแกนบาร์โค้ด และกดขายเพื่อสร้างบิลแบบชั่วคราวได้ โดยรายการเหล่านี้จะเข้าคิวรอส่งขึ้นระบบหลังบ้านทันทีที่ระบบเชื่อมต่ออินเทอร์เน็ตกลับมาใช้งานได้ปกติ
*   **จำกัดการทำงาน:** จะทำการปิดกั้นฟังก์ชันงานที่ไม่ปลอดภัยเมื่อ Offline เช่น การตรวจสอบยอดเงินในกระเป๋าเงินสมาชิก (เพื่อป้องกันปัญหาการใช้เงินซ้ำซ้อน) และการโอนสต็อกสินค้าข้ามสาขา

### 6. ตารางเปรียบเทียบสิทธิ์และตัวชี้วัด (UAT & Verification Metrics)

| คุณสมบัติที่ต้องการทดสอบ | วิธีการตรวจสอบ | ผลลัพธ์ที่ยอมรับได้ (Criteria) |
|---|---|---|
| **Data Leak Prevention** | ทำการจำลองส่ง API Requests ไปขอข้อมูล Order ของร้านค้า B โดยใช้ User Token ของร้านค้า A | ระบบต้องปฏิเสธการเข้าถึงด้วย HTTP 403 Forbidden เสมอ |
| **Immutable Snapshot** | ลองทำรายการขายบริการ "สปาผม" ราคา 500 บาท จากนั้นเข้าไปที่ตั้งค่าบริการเพื่อแก้ไขราคาเป็น 600 บาท และเปลี่ยนชื่อบริการ แล้วกดเปิดรายงานออเดอร์เก่า | รายงานของออเดอร์เก่าต้องโชว์ราคา 500 บาทและแสดงชื่อบริการแบบดั้งเดิม |
| **Race Condition Stock** | ส่งออเดอร์สั่งซื้อสินค้าชิ้นเดียวกันพร้อมกัน 5 ออเดอร์จากอุปกรณ์ต่างกันในวินาทีเดียวกันขณะสต็อกเหลือ 1 ชิ้น | ระบบต้องตัดผ่านได้เพียงออเดอร์เดียว อีก 4 ออเดอร์ต้องถูก Reject ด้วย "สต็อกสินค้าไม่พอ" และยอดการบันทึกใน StockMovement ต้องติดลบได้ถูกต้องตรงตามเหตุการณ์จริง |
